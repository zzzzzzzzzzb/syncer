# 局域网多端剪切板同步系统架构方案（Rust Core）

## 1. Summary
- 目标：先交付“局域网内实时剪切板同步”MVP，后续平滑扩展到“文件同步”和“服务端托管模式”。
- 关键约束：Rust 负责核心逻辑；客户端需覆盖 Windows/macOS/iOS/Android；当前阶段仅输出架构设计，不生成代码。
- 已确认决策：全平台并行推进；设备发现采用 mDNS 零配置；安全基线采用端到端加密 + 配对码。
- 客户端框架选型：采用 Flutter 作为统一客户端框架，Rust 通过 FFI 提供核心能力。

## 2. Current State Analysis
- 仓库现状：`f:\project\rust\syncer` 为空目录，无现有源码、配置、构建脚手架或入口文件。
- 影响评估：
  - 无历史包袱，适合按“领域边界优先”设计清晰分层。
  - 需要先定义模块边界、协议、错误模型和测试策略，再进入实现。
  - 由于全平台并行，必须在初期避免 UI 与同步核心耦合，确保 Rust Core 可复用。

## 3. Assumptions & Decisions

### 3.1 产品范围决策
- 初期范围（In Scope）：
  - 局域网设备自动发现。
  - 双向实时剪切板文本同步（UTF-8 文本为首期强约束）。
  - 安全配对、会话加密、设备信任管理。
  - 基础冲突处理（同一时间多端更新时的去重与最终一致）。
- 非初期范围（Out of Scope）：
  - 文件/目录同步。
  - 云端账号、远程中继、离线消息托管。
  - 富文本/图片等复杂剪切板格式（作为后续扩展）。

### 3.2 客户端技术选型决策
- 选择：Flutter（UI + 跨平台壳） + Rust Core（业务与网络）。
- 选型理由：
  - 单代码栈覆盖桌面与移动，满足全平台并行要求。
  - Rust 核心逻辑可高度复用，降低跨端行为不一致风险。
  - 与 Rust 的桥接生态成熟（如 flutter_rust_bridge 类方案），可隔离 FFI 复杂度。
- 备选未采用：
  - Tauri：移动端链路与工程复杂度在当前目标下不优。
  - React Native：Rust 深度集成与多端系统能力收敛成本偏高。

### 3.3 架构决策
- 架构风格：分层 + 端到端协议驱动。
- 核心原则：
  - “同步状态机”仅在 Rust Core 内实现，客户端仅消费事件流。
  - “发现/连接/配对/传输/存储”解耦为独立模块，便于后续替换为托管模式。
  - 所有跨端能力通过稳定接口暴露，避免 UI 直接依赖网络细节。

## 4. Proposed Changes

> 说明：以下为后续实施时将创建的目标结构与职责定义（当前不执行代码创建）。

### 4.1 仓库结构（目标）
- `crates/syncer-core`：领域核心与会话状态机（无平台依赖）。
- `crates/syncer-discovery`：mDNS 广播与发现抽象。
- `crates/syncer-transport`：安全通道、消息编解码、重连与心跳。
- `crates/syncer-clipboard`：平台剪切板适配抽象（trait + 各端实现挂载点）。
- `crates/syncer-ffi`：对 Flutter 暴露稳定 FFI API（命令调用 + 事件订阅）。
- `apps/flutter_client`：Flutter 多端客户端（统一 UI、配对流程、设备管理）。
- `docs/architecture`：架构说明、协议文档、威胁模型、演进路线。

### 4.2 核心数据流（MVP）
1. 设备启动后通过 mDNS 广播能力与监听局域网服务。
2. 用户在任一设备输入配对码，双方完成密钥协商并建立信任关系。
3. Rust Core 监听本地剪切板变更，生成事件（含设备ID、时间戳、版本向量）。
4. 事件经加密通道发送到已配对设备，远端通过去重策略决定是否应用。
5. 应用成功后回传确认，状态机更新会话状态与最后同步点。

### 4.3 协议与接口决策
- 会话消息类型：
  - `Hello`（能力协商）
  - `PairInit/PairConfirm`（配对握手）
  - `ClipboardUpdate`（增量更新）
  - `Ack`（确认）
  - `Heartbeat`（保活）
- 可靠性策略：
  - 至少一次投递 + 幂等去重（事件ID + 版本信息）。
  - 心跳超时触发重连与状态恢复。
- FFI 接口形态（高层）：
  - 命令型：启动服务、开始配对、查询设备、断开连接。
  - 事件型：发现设备、配对状态变化、同步成功/失败、冲突通知。

### 4.4 安全方案（MVP 基线）
- 配对阶段：用户可见配对码用于短时认证绑定。
- 传输阶段：全链路加密，未配对设备不可读取有效载荷。
- 信任管理：本地保存受信设备指纹；支持撤销与重新配对。
- 防护重点：重放防护（nonce/计数器）、会话过期、异常断线后的密钥更新策略。

### 4.5 可扩展性预留（面向后续）
- 文件同步扩展位：
  - 协议新增 `FileOffer/FileChunk/FileAck`，复用会话层与安全层。
  - 传输层支持分片、断点续传、校验和。
- 托管模式扩展位：
  - 发现层抽象出 `DiscoveryProvider`，新增 `CloudRegistryProvider` 即可切换。
  - 传输层支持“直连优先 + 中继回退”策略。

## 5. Implementation Steps
1. 明确并固化领域边界：核心状态机、发现、传输、剪切板、FFI 五大模块。
2. 完成协议草案：消息模型、握手流程、错误码与重试语义。
3. 完成安全设计：配对码生命周期、密钥协商、信任存储与吊销机制。
4. 完成客户端交互设计：配对、设备列表、同步状态、异常提示。
5. 制定测试矩阵：跨平台组合、网络抖动、冲突写入、重连恢复、安全异常。
6. 定义里程碑：M1（单链路可用）→ M2（多设备稳定）→ M3（跨平台联调完成）。

## 6. Verification Steps
- 架构完整性检查：
  - 是否覆盖发现、配对、安全、同步、冲突、重连全链路。
  - 是否存在模块职责交叉或循环依赖风险。
- 可实现性检查：
  - 每个模块是否可独立开发与测试。
  - FFI 接口是否足够支撑 Flutter 客户端功能闭环。
- 可演进性检查：
  - 文件同步是否仅需新增消息类型与传输能力，不破坏现有协议。
  - 托管模式是否可通过替换发现/路由提供者完成。
- 验收标准（本次规划输出）：
  - 已给出可执行的模块边界、协议方向、安全基线、测试与演进路径。
  - 执行者可按本方案直接进入工程初始化与实现阶段，无需再做关键技术选型决策。
